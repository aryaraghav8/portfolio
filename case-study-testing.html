01 — The Problem

Three problems colliding at once

The websites were being shipped without a structured QA process. Bugs were slipping through to production, cross-browser inconsistencies were breaking layouts for real users, and UI issues were eroding trust. These weren't isolated problems — they were symptoms of a missing quality gate.

🚨
Bugs in Production

No formal QA process meant broken features and layout issues were reaching live users before being caught and fixed.

🌐
Cross-browser Failures

Pages rendered fine in Chrome but broke in Safari, Firefox, and Edge — affecting a significant portion of real users.

🔧
Unresolved UI Issues

Misaligned elements, broken hover states, and usability friction were accumulating with no systematic way to track or fix them.

02 — What I Found

The bugs I caught

Systematic testing across browsers and devices uncovered a range of issues — from critical layout breaks to usability friction. Every issue was documented, prioritised, and resolved.

03 — The Process

Building a quality gate

01
Cross-browser Test Matrix

Created a structured matrix covering Chrome, Firefox, Safari, and Edge across desktop and mobile. Every major page was tested systematically.

02
Bug Documentation & Prioritisation

Documented every issue with screenshots, browser/device context, and severity level. Prioritised critical layout breaks over cosmetic issues.

03
Frontend Fixes & CSS Patches

Applied targeted CSS fixes — vendor prefixes for Safari, flexbox fallbacks for Firefox, layout patches for Edge — without causing regressions elsewhere.

04
Usability Improvements

Beyond bugs — identified unclear CTAs, missing focus states, and hard-to-tap mobile buttons, and fixed these alongside the technical issues.

05
Regression Testing & Sign-off

Re-tested every fix across the full browser matrix before marking issues resolved. Delivered a final QA report with notes and recommendations.

04 — Impact

Quality that shipped

Establishing a QA process changed the way the team thought about quality — from reactive bug-fixing to proactive issue prevention.

05 — Learnings

What I took away

QA is a design responsibility too. Catching usability issues during testing — not just technical bugs — is something designers are uniquely positioned to do, and it significantly raises overall product quality.

A structured test matrix removes ambiguity. Without a systematic approach, testing becomes random and gaps are inevitable. The matrix ensured nothing fell through the cracks.

Cross-browser issues are predictable once you know the patterns. Safari's WebKit quirks, Firefox flexbox behaviour, and Edge CSS handling each have known failure modes — learning these once saves hours every project.
